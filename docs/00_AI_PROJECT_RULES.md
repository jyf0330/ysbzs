# AI 项目总规则 · 元素背包史

## 项目标识

- 项目名称：元素背包史（ysbzs）
- 项目类型：`game`
- 工作流：`ywh-game`
- 仓库：`git@github.com:jyf0330/ysbzs.git`
- 主文件：`index.html`（单文件原型，禁止无故拆分）
- 测试命令：`node test.js`（当前基准：156/156）

## 入口规则

1. 任何任务开始前必须调用并阅读 `ywh-game` skill。
2. 必须运行 `git status --short --untracked-files=all`，区分历史改动和本轮改动。
3. 必须按本文件的分类表判断任务类型。
4. 除非任务极小且无需交接，否则先创建或更新 `tasks/doing/当前任务.md`。

## 固定口令：同步 ywh 工作流

当用户说“同步 ywh 工作流”“同步当前项目的 ywh 工作流规则”“以上游 ywh / ywh-game 为准同步项目规则”时，含义固定为：

- 以上游 `ywh` / `ywh-game` skill、references 模板和 registry 为准。
- 只同步项目工作流结构与 AI 入口，不改 `index.html`、`test.js` 或游戏核心代码。
- 检查并更新 `docs/00_AI_PROJECT_RULES.md`、`AGENTS.md`、`CLAUDE.md`、`.github/copilot-instructions.md`。
- 必要时补齐或更新 `docs/00_CURRENT_CONTEXT.md`、`docs/09_DECISIONS.md`、`tasks/doing/当前任务.md`。
- 同步内容限定为任务分类、Demo 阶段文档门禁、按需补充文档、任务卡分流、最小读取范围、验收规则、归档文档定位、CHANGELOG/Git 收尾规则。
- 如发现项目规则与上游规则冲突，先输出冲突文件、冲突内容、可能影响和推荐方案，再决定是否覆盖。
- 验证使用 `rg` 引用搜索、模板存在性检查、同步映射一致性检查；没有游戏运行改动时不强制运行游戏测试。

## 任务分流与最小读取

不要让所有 AI 读所有文件。正确流程是：总控 AI 先判断任务类型并生成任务卡；执行 AI 只读任务卡、当前上下文、相关 2-4 份文档和指定代码文件；验收 AI 只读任务卡、验收清单和测试结果。

### 通用读取上限

- 1 个任务卡：`tasks/doing/当前任务.md`
- 1 个当前上下文：`docs/00_CURRENT_CONTEXT.md`
- 2-4 个相关文档
- 1-3 个相关代码文件

超过这个范围，先拆任务，不要默认全量读取 `docs/`。

### 多 AI 协作保护

- 一个任务只允许一个 AI 修改同一代码文件。
- 其他 AI 只能审查、验收或修改任务卡明确授权的文档。
- 执行 AI 不允许自行扩大任务范围。
- 归档文档不得作为当前规则源头。
- 如发现冲突，先输出冲突文件、冲突内容、可能影响和可选方案，再等用户决策。

## graphify 使用边界

`graphify` 只用于总控 AI 做项目地图和影响范围分析，不作为执行 AI 默认全量读取项目的理由。

适用场景：

- 新接手项目或首次梳理大型目录。
- 大需求 / 核心战斗规则改动。
- 架构梳理、核心/展示边界检查。
- 文档体系治理或同步 `ywh` 工作流。
- 不确定一次改动会影响哪些代码、测试和文档。

使用规则：

- 总控 AI 可用 `graphify` 辅助判断读取范围，并把结论压缩进任务卡。
- 执行 AI 仍然只能读取任务卡指定文件。
- 不得因为有图谱就全量读取 `docs/`、`index.html` 或归档文档。
- 修改代码前必须读取真实源码，不能只凭图谱修改。
- 图谱与当前源码或正式文档冲突时，以当前源码和正式文档为准。

## 变更分类

| 分类 | 判断标准 | 走哪条路 |
|---|---|---|
| 大需求 / 新模块 | 新系统、新玩法机制、架构重构、核心战斗规则变更 | 文档门禁 → 补草稿 → 用户确认 → 实现 |
| 小修改 / 局部补丁 | 纯 bug、数字调参、非核心 UI 微调、不改玩法规则 | 最小实现 → 验证 → 同步文档 |
| 排查 / 修 bug / 验收 | 复现问题、读日志、验收测试 | 先复现 → 最小修复 → 验证 |
| 工作流规则变更 / 文档治理 | 修改 `ywh` / `ywh-game` / 模板 / registry / AI 规则 / 文档结构，不改变游戏运行行为 | 最小规则修改 → 引用搜索 / 模板存在性 / 同步映射一致性验证 → 同步 CHANGELOG |

代码量小不等于小修改。凡涉及回合结构、伤害时序、引爆逻辑、元素规则、状态机等核心战斗机制，无论代码量，一律视为大需求。

## 正式文档与归档

- `docs/99_归档/` 只保存历史草稿，不是当前规则源头。
- 根目录不长期存放 `main.txt`、`srd.txt`、`数据.txt`、`验收结果.txt` 这类草稿。

- `docs/09_DECISIONS.md` 记录已拍板的长期协作决策，避免 AI 反复争论同一问题。

## 实现规则

- 修改 `index.html` 前先读相关上下文。
- 每次改动最小化，不过度工程化。
- 不引入新依赖，除非用户明确要求或当前任务无法完成。
- 不添加无关注释、docstring 或新抽象。
- 不自动拆分单文件架构。

## 验证规则

- 改游戏运行行为：运行 `node test.js`，必要时做 Playwright / 截图 / console error 检查。
- 工作流规则变更 / 文档治理：运行引用搜索、模板存在性、同步映射一致性检查；不要求游戏运行测试，但可用 `node test.js` 复核基线。
- 测试失败时不进入 Git 提交。

## 代码与文档不一致处理

如果发现实际代码和现有文档不一致，必须先输出「代码-文档差异清单」，至少包含：

- 涉及的代码文件
- 涉及的文档文件
- 不一致点
- 推荐以代码为准还是以文档为准

然后只允许给用户这 3 个选择：

1. 以代码为准，更新文档
2. 以文档为准，修改代码
3. 逐项决定，按差异清单分别处理

在用户做出选择前，不得擅自同时改代码和文档把分歧抹平。

## Git 规则

- 有效改动必须提交，且不要把历史未提交的无关改动混入本轮。
- commit 格式优先使用 Lore Commit Protocol；简短提交至少包含 `docs:` / `feat:` / `fix:` 等类型。
- 安全检查通过后默认 `git push`。
- 禁止 push `.env`、密钥、`node_modules`、大体积临时文件。
- 禁止 force push 到主分支。

## 禁止事项

- 禁止跳过 `ywh-game` skill。
- 禁止跳过变更分类。
- 禁止所有 AI 默认全量读取 `docs/`。
- 禁止多个 AI 同时修改同一代码文件。
- 禁止把归档草稿当成当前规则。
- 禁止把「代码量小」当作「不同步文档」的理由。
- 禁止提交后不验证。
