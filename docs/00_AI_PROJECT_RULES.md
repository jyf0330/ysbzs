# AI 项目总规则 · 元素背包史

## 项目标识

- 项目名称：元素背包史（ysbzs）
- 项目类型：`game`
- 工作流：`ywh-game`
- 仓库：`git@github.com:jyf0330/ysbzs.git`

## Step 0：强制调用 ywh-game skill（不可跳过）

**任何任务开始前，AI 必须先调用并阅读 `ywh-game` skill，再决定走哪个分支。**  
跳过此步视为违规，不得进入实现。

## Step 1：变更分类（必须在实现前完成）

按以下标准判断，选一个分支：

| 分类 | 判断标准 | 走哪条路 |
|---|---|---|
| **大需求** | 新系统 / 新玩法机制 / 架构重构 / 核心战斗规则变更 | 文档门禁 → 补草稿 → 用户确认 → 实现 |
| **小修改** | 纯代码 bug / 数字调参 / UI 微调 / 不改玩法规则 | 直接实现 → 验证 → 同步文档 |
| **排查/验收** | 复现问题 / 读日志 / 验收测试 | 先复现 → 最小修复 → 验证 |
| **工作流规则变更 / 文档治理** | 修改 `ywh` / `ywh-game` / 模板 / registry / AI 规则文件，不改变游戏运行行为 | 最小规则修改 → 引用搜索 / 模板存在性 / 同步映射一致性验证 → 同步 CHANGELOG |

> **注意**：改动代码量小 ≠ 小修改。  
> 凡涉及「回合结构、伤害时序、引爆逻辑、元素规则、状态机」等核心战斗机制，无论代码量，一律视为**大需求**。

## 实现规则

- 单文件架构：所有逻辑在 `index.html` 中，禁止无故拆分。
- 修改前先读文件，理解上下文后再改。
- 每次改动最小化，不过度工程化。
- 不添加无关注释、docstring、类型注解。

## 测试规则

- 测试文件：`test.js`（Node.js，`node test.js`）。
- 每次改动后运行 `node test.js`，确保全部通过（当前基准：151 项）。
- 测试失败时不进入文档同步和 Git 提交。

## Web-Game 验证规则

有意义改动后必须：

1. 运行 Playwright 脚本或 action_payloads 验证游戏行为。
2. 检查截图和 console error。
3. 用 `render_game_to_text` 输出游戏状态。
4. 验证通过后再进入文档同步。

## 小修改规则

1. 定位相关文件，做最小修改。
2. 运行 `node test.js` 验证。
3. 验证通过后同步相关文档。
4. 更新 `docs/10_CHANGELOG.md`。
5. Git commit + push。

## 文档同步规则

- 改玩法 / 数值 / 规则 → 更新 GDD 或功能拆解。
- 改代码结构 / 模块边界 / 数据结构 → 更新技术架构总览。
- 改 UI / HUD / 菜单 / 界面信息层级 → 更新 `HUD与界面信息设计.md`。
- 改输入操作 / 按钮行为 / 模式切换 / 反馈规则 → 更新 `用户界面与操作规范.md`。
- 改美术风格 / 资源路径 / 资源命名 → 更新美术文档。
- 改验收标准 → 更新版本发布验收清单。

## Web-Game 验证规则

2. 检查截图和 console error。

## 小修改完整流程

1. 调用 ywh-game skill，分类确认为「小修改」。
2. 定位文件，做最小修改。
3. 运行 `node test.js` 验证（全部通过）。
4. 按文档同步表更新相关文档。
5. 更新 `docs/10_CHANGELOG.md`。
6. Git commit + push。

## 小修改防膨胀规则

- 不为了一个局部改动重梳整个项目。
- 不生成大文档，不补齐无关文档包。
- 不引入新抽象或新依赖，除非本次修改无法完成。
- 不修改无关文件。

## Bug / 验收规则

1. 先复现问题（读代码、读测试、读日志）。
2. 定位根因，做最小修复。
3. 运行 `node test.js` + Playwright 验证。
4. 更新相关文档和 CHANGELOG。
5. Git commit + push。

## 代码与文档不一致处理

如果发现实际代码和现有文档不一致，必须先输出「代码-文档差异清单」，至少包含：

- 涉及的代码文件
- 涉及的文档文件
- 不一致点
- 推荐以代码为准还是以文档为准

然后只允许给用户这 3 个选择：

1. 以代码为准，更新文档
2. 以文档为准，修改代码
3. 逐项决定，按差异清单分别处理

在用户做出选择前，不得擅自同时改代码和文档把分歧抹平。

## Git 规则

- 有改动必须 `git add .` → `git commit`。
- commit 格式：`<type>: <中文摘要> / <English summary>`。
- type 可用：`feat` / `fix` / `refactor` / `test` / `docs` / `chore`。
- 安全检查通过后默认 `git push`。
- 禁止 push `.env`、密钥、`node_modules`、大体积临时文件。
- 无 remote 时输出 `PUSH_SKIPPED_NO_REMOTE`。

## 禁止事项

- 禁止跳过 Step 0（调用 ywh-game skill）。
- 禁止跳过 Step 1（变更分类）直接实现。
- 禁止绕过文档门禁。
- 禁止把「代码量小」当作「小修改」的依据——要看是否影响游戏机制。
- 禁止自动大范围重构。
- 禁止把游戏项目当工具项目写。
- 禁止 force push 到主分支。
- 禁止提交后不验证。
- 禁止改完不同步对应文档。
