## 1. 设计目标

商店不是“买东西页面”，而是“每回合构筑决策页面”。

核心体验目标：

1. 构筑深度：玩家每轮都在判断同名升级、转派、刷新与冻结的机会成本。
2. 紧张抉择：有限金币、有限格子、有限回合内形成高价值决策。
3. 快节奏可读：在 45-90 秒内完成一次高质量商店决策。

情绪锚点对齐：

- 参考《The Bazaar》的回合推进感与成长预期。
- 参考《Super Auto Pets》的同名合成爽点与买卖节奏。
- 参考《杀戮尖塔》的“每次选择影响后续”压力结构。

---

## 2. 设计原则（Pillars）

1. 一拍一跃迁：每次购买/合成后，玩家必须立刻感知变强。
2. 转派可计算：转流派有代价，但代价必须可读、可解释、可预期。
3. 连锁优先：奖励成体系的连锁构筑，而非无脑堆高稀有度。
4. 高密度可读：信息多，但关键决策信息始终位于第一视线层。

反支柱（Anti-pillars）：

1. 不做大而全经营看板（避免稀释每轮决策张力）。
2. 不做隐藏概率黑箱（避免破坏挑战与精通）。
3. 不做纯稀有度碾压（避免削弱同名合成价值）。

---

## 3. 页面结构与信息层级

## 3.1 结构分区

建议三栏结构（桌面端）：

1. 左栏（决策摘要栏）：
- 当前金币
- 当前天数/阶段
- 本轮关键提示（如“可合成”“可转派”“金币紧张”）
- 本轮目标流派标签

2. 中栏（商店商品主区）：
- 单位卡列表（主焦点）
- 每张卡的等级、价格、元素、核心标签
- 冻结态显示
- 购买按钮与禁用态

3. 右栏（阵容与后效栏）：
- 上阵位（active）
- 备战位（bench）
- 同名进度提示（例如 2/3）
- 购买后预估变化（战力、连锁、成型度）

底部固定操作条：

- 刷新（显示成本）
- 完成商店
- 可选的“本轮回顾”入口（仅原型期）

## 3.2 优先级层

P0（第一眼必须看到）：金币、商品价格、是否可买、是否可合成、刷新成本。

P1（一轮决策必须看到）：
- 同名进度
- 上阵/备战空位
- 购买后预估变化
- 冻结状态

P2（有空再看）：
- 详细描述文案
- 扩展提示
- 历史回放

## 4. 单位卡设计规范

每张单位卡最少包含：

1. 识别区：
- 名称
- 等级/品级
- 元素图标

2. 价值区：
- 价格
- 是否可买（可用/禁用）
- 冻结状态

3. 决策区：
- 同名进度（例如“同名 2/3”）
- 购买后收益摘要（简短文本）
- 转派关联提示（例如“偏离当前核心”）

4. 动作区：
- 购买
- 冻结/解冻

反馈要求：

- 购买成功：卡片离场 + 阵容栏即时变化 + 日志一句话。
- 合成触发：有明显动画/强调色/音效（原型可先做文案与高亮）。
- 金币不足：按钮禁用 + 点击提示原因。

## 5. 关键交互流程

## 5.1 购买流程

1. 玩家点击购买。
2. 系统校验金币与格位。
3. 扣费并入阵（上阵优先，其次备战）。
4. 若同名满足条件，触发合成。
5. 更新所有关联 UI（金币、阵容、同名进度、日志）。

## 5.2 冻结流程

1. 玩家点击冻结。
2. 卡片进入冻结态并强化边框标识。
3. 刷新后冻结卡保留在原位。
4. 关店后是否保留冻结，按当前玩法版本规则执行（实现与文档需保持一致）。

## 5.3 刷新流程

1. 点击刷新。
2. 扣除刷新费用。
3. 非冻结卡重置为新商品。
4. 保留冻结卡，更新可买状态。

## 5.4 卖出流程

1. 玩家在阵容区点击出售。
2. 返还等级对应金币。
3. 清理槽位并更新相关显示。

## 5.5 完成商店

1. 玩家点击“完成商店”。
2. 商店关闭并进入下一阶段。
3. 记录本轮关键决策日志（原型期建议保留）。

---

## 6. 状态与文案规范

必须区分并可见：

1. 可购买
2. 金币不足
3. 冻结
4. 已售罄
5. 可合成
6. 可转派（提示）

建议文案短句：

- 可合成："可合成：再买 1 张升级"
- 转派风险："转派将延后当前成型"
- 刷新提醒："刷新将消耗 X 金币"

---

## 7. Prototype-First 验证口径

沿用 `reports/playtest/h5-shop-validation-sheet.md`，本界面设计需满足：

1. 购买后 1 秒内看懂收益变化。
2. 同名合成行为形成主动追牌。
3. 刷新/冻结/转派均可口述决策理由。
4. 单轮商店操作在 45-90 秒完成。

关键指标：

- 决策清晰度平均 >= 7/10
- 爽感平均 >= 7/10
- 构筑路线形成率 >= 80%（5 局中至少 4 局）

---

## 8. 交付边界

本轮仅定义 UI/UX 与验证口径，不含：

1. 商店核心规则重构
2. 价格与经济数值重平衡
3. 代码实现细节与测试用例编写

后续实现任务需单独开卡，并按 TDD 执行。

## 9. 实现对接清单（给程序）

1. UI 结构：三栏布局 + 底部操作条。
2. 卡片字段：名称、等级、价格、元素、同名进度、状态标签。
3. 实时反馈：购买、合成、冻结、刷新、卖出全部有明确回执。
4. 日志最小集：本轮至少记录 1 条关键决策语句。
5. 验证入口：保留快速运行 H5 验证记录的可见入口（文案或按钮均可）。
