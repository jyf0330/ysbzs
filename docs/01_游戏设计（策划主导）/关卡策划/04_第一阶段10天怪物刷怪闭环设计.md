## 设计前提

- 当前代码：5 天 run 扩展为 10 天。
- 玩家作战路线：**火伤害闭环 + 中立召唤流闭环**。召唤流不绑定水系。
- 怪物特殊机制和英雄特殊能力归入统一 ability system，不走 if/else 散写。
- 怪物数值全为具体值。所有总HP/总ATK 由怪物类型表自动计算，不手填。
- Day1-2 不过难。

## 一、统一 Ability System 接口（预留）

所有怪物能力和英雄被动/主动技能走同一套能力框架。

### 能力钩子（Hooks）

| Hook | 触发时机 | 用途示例 |
|------|---------|---------|
| onRoundStart | 回合开始时 | 熔岩涌动(铺火) |
| onMoveIntent | 移动意图时 | 堵路不绕路 |
| onChooseTarget | 选择攻击目标时 | 优先攻城堡、护卫光环 |
| onBeforeDealDamage | 造成伤害前 | 伤害倍率 |
| onBeforeTakeDamage | 受伤前 | 铁壁减伤 |
| onAfterTakeDamage | 受伤后 | 效果计数 |
| onDeath | 死亡时 | 死亡留水、死亡铺火 |
| onSpawn | 生成时 | 召唤分裂 |
| onEveryNthRound | 每N回合 | 分裂召唤 |

### 第一阶段执行规则

| 能力类型 | 状态 | 说明 |
|---------|------|------|
| 普通移动 | 启用 | 现有寻路逻辑 |
| 普通攻击 | 启用 | 现有攻击逻辑 |
| HP/ATK/AP 数值 | 启用 | 决定击杀速度和防线压力 |
| 怪物数量压力 | 启用 | 通过 budget/spawnSize 控制 |
| Boss 高HP/高ATK | 启用 | Boss 作为高数值单位 |
| 已有火层伤害 | 启用 | fire_starter/ember 等基础火英雄 |
| 基础召唤 | 启用 | sprout_summoner 基础召唤 |
| 召唤物普通行动 | 启用 | runSummonActions 普通攻击 |
| 英雄特殊被动(全部) | PENDING | ember_seed/fluff_speaker/boom_sprite 等 |
| 堵路不绕路(blocker) | PENDING | ability 占位 |
| 优先攻城堡(siege) | PENDING | ability 占位 |
| 铁壁减伤(boss5) | PENDING | ability 占位 |
| Boss护卫光环(minion) | PENDING | ability 占位 |
| 自身铺火(boss8) | PENDING | ability 占位 |
| 每2回合召唤(boss10) | PENDING | ability 占位 |

## 三、Ability 配置示例

```js
// 怪物 ability 字段示例（存入 MONSTER_TYPES）
const MONSTER_TYPES = {
  blocker: { name:'堵路怪', hp:14, atk:1, ap:3,
    ability: { id:'block_path', trigger:'onMoveIntent',
      status:'pending',
      description:'前方有单位时不绕路，直接停下' }
  },
  siege: { name:'攻城怪', hp:7, atk:2, ap:5,
    ability: { id:'target_castle', trigger:'onChooseTarget',
      status:'pending',
      description:'优先攻击城堡' }
  },
  boss5: { name:'炎甲兽', hp:35, atk:4, ap:4,
    ability: { id:'iron_wall', trigger:'onBeforeTakeDamage',
      status:'pending',
      description:'单次受伤>=5时减伤2(当前回合)' }
  },
  boss8: { name:'熔岩核心', hp:45, atk:4, ap:5,
    ability: { id:'lava_surge', trigger:'onRoundStart',
      status:'pending',
      description:'自身位置生成1层火' }
  },
  boss10: { name:'远古炎核', hp:60, atk:5, ap:5,
    ability: { id:'core_split', trigger:'onEveryNthRound',
      config:{n:2}, status:'pending',
      description:'每2回合召唤1只normal(hp=4,atk=1)' }
  },
};
```

（第2版使用怪物类型表自动计算总HP/总ATK，以下分析基于修正后数据）

**Day5后半段起压力显著偏高**：Day5全天HP=216, ATK=33, Day7全天HP=306, ATK=46。
这些数值比第1版手算值高50%-100%，原因是之前严重低估了精英/堵路/攻城/Boss的HP叠加。

这是整个10天最难的一天（Day10反而低一些：HP=386, ATK=59, 因为数量少但单体超强）。

## 八、审核重点

后续审核不是删怪物机制，而是看以下6点：

1. **怪物基础数值能否跑完10天**：只用HP/ATK/AP达成难度曲线。
2. **怪物ability占位是否清楚**：每个怪物ability字段有id/trigger/status/description。
3. **哪些能力第一阶段启用,哪些PENDING**：按执行规则标注清楚。
4. **商店成长是否能应对当天怪物**：Day2买第3英雄, Day4召唤系, Day5 T3, Day7 T4, 商店供给是否匹配。
5. **能力系统接口是否同时支持英雄和怪物**：9个hooks能否覆盖两者。
6. **数值风险**：§七当前总HP偏高, 需在实现前决定是否降数值。
