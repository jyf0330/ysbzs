> 日期：2026-05-28
> 来源：ywh 架构设计

## 一、大流程状态机（一天怎么走）

```
DayStart
  ↓
MorningBattle
  ↓
MiddayShop
  ↓
AfternoonBattle
  ↓
NightShop
  ↓
DayEnd
  ↓
NextDayStart
```

### 各阶段职责

| 阶段 | 做什么 | 关键规则 |
|---|---|---|
| DayStart | 城堡HP回满100/100 | 城堡不跨天 |
| MorningBattle | 验证昨晚买的构筑 | 标准战斗流程 |
| MiddayShop | 补英雄、刷新、强化流派 | 玩家成长在此发生 |
| AfternoonBattle | 验证中午新构筑 | 标准战斗流程 |
| NightShop | 埋新机制（如第1晚出召唤使） | 为下一天准备 |
| DayEnd | 结算奖励，保留英雄/金币/等级 | 玩家成长跨天 |
| NextDayStart | 城堡回满，敌人升级 | 新一轮 |

### 铁律

> **城堡血量不跨天，玩家成长跨天。**

## 二、战斗状态机（一回合怎么结算）

```
BattleStart
  ↓
PlayerPlan
  ↓
HeroAction       ← 只叠元素层，不扣血
  ↓
SummonAction     ← 召唤物从元素格出生
  ↓
ElementResolve   ← 统一结算所有元素伤害
  ↓
MonsterAction    ← 活着的怪物行动
  ↓
CheckBattleEnd
  ↓
NextRound / BattleEnd
```

### 各阶段详解

#### BattleStart
生成怪物、城堡、英雄、已有召唤物。

#### PlayerPlan
玩家拖英雄、改方向、看预览。不扣血，只显示预判。

#### HeroAction
**英雄只叠层，不立刻扣血。**

```
火苗使 第1击 → 怪物叠火2层
火苗使 第2击 → 怪物叠火4层
火苗使 第3击 → 怪物叠火6层
```

#### SummonAction
召唤物行动。**和元素格强关联：**

| 所在格 | 消耗 | 生成 | HP | ATK | 特效 |
|---|---|---|---|---|---|
| 空白格 | 无 | 普通小灵 | 6 | 1 | — |
| 火3格 | 火3层 | 火灵 | 9 | 1 | 攻击叠火1层 |
| 水3格 | 水3层 | 水灵 | 9 | 1 | 攻击叠水1层 |
| 风3格 | 风3层 | 风灵 | 9 | 1 | 攻击叠风1层 |
| 土3格 | 土3层 | 土灵 | 9 | 1 | 攻击叠土1层 |

> 这让铺元素有第二种用途：爆炸，或者转化成召唤物。

#### ElementResolve
统一结算所有元素伤害。火6层=21伤害，水3层=6伤害，etc。

#### MonsterAction
活着的怪物移动→攻击英雄/召唤物/元素格→触发反伤。

#### CheckBattleEnd
结束条件：怪物全清、城堡归零、英雄全灭、达到回合上限。

## 四、和当前 index.html 的差距

### 4a. 当前 phase 只有 4 个值

```js
phase: 'PLAYER' | 'MONSTER' | 'SHOP' | 'OVER'
```

### 需要扩展为

```js
macroPhase: 'DAY_START' | 'MORNING_BATTLE' | 'MIDDAY_SHOP' |
            'AFTERNOON_BATTLE' | 'NIGHT_SHOP' | 'DAY_END'

battlePhase: 'PLAN' | 'HERO_ACTION' | 'SUMMON_ACTION' |
             'ELEMENT_RESOLVE' | 'MONSTER_ACTION' | 'BATTLE_END'
```

### 4b. 当前 endPlayerTurn 做的事

```js
endPlayerTurn():
  1. 执行未用槽位
  2. settleExplosions()
  3. monsterAct()
  4. 检查回合结束 → 商店或下一波
```

所有东西揉在一起。需要拆成独立阶段。

### 4c. 当前缺少的阶段

| 阶段 | 当前状态 |
|---|---|
| PlayerPlan | 🟡 有预览系统但无正式"规划阶段" |
| SummonAction | ❌ 完全不存在 |
| ElementResolve | 🟡 settleExplosions 存在但混合在 endPlayerTurn |
| MonsterAction | ✅ monsterAct 存在 |
| CastleReset | ❌ 城堡每天回满不存在 |
| DayEnd | ❌ 不存在 |
| NextDayStart | ❌ 不存在 |

### 4d. 关键顺序保证

当前代码 `endPlayerTurn` 里：
```js
commitPlayerActionsToElementField()  // 叠层写入
settleExplosions()                    // 结算
monsterAct()                          // 怪物行动
```

叠层→结算 顺序是对的。但缺少：
- 召唤物阶段（在叠层之后、结算之前）
- 明确的 PlayerPlan 阶段（当前是边看边操作）

## 五、建议实现顺序

### P0：重构战斗状态机

不改任何游戏逻辑，只把 `endPlayerTurn` 拆成独立阶段：

```js
function endPlayerTurn() {
  // Phase 1: HERO_ACTION
  processHeroActions();      // 提交英雄行动槽

  // Phase 2: SUMMON_ACTION  （先留空，后续填）
  // processSummonActions();

  // Phase 3: ELEMENT_RESOLVE
  settleExplosions();         // 统一结算

  // Phase 4: MONSTER_ACTION
  monsterActAll();            // 怪物行动

  // Phase 5: BATTLE_END
  checkBattleEnd();
}
```

### P1：加大流程状态机

```js
function advanceMacroPhase() {
  switch(G.macroPhase) {
    case 'DAY_START':   startMorningBattle(); break;
    case 'MORNING_OVER': openMiddayShop();     break;
    case 'MIDDAY_OVER':  startAfternoonBattle(); break;
    case 'AFTERNOON_OVER': openNightShop();    break;
    case 'NIGHT_OVER':   endDay();             break;
  }
}
```

### P2：召唤物系统

在 SUMMON_ACTION 阶段实现召唤→消耗元素→生成宠物。
