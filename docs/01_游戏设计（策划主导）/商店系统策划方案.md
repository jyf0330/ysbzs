## 1. 商店核心循环

### 1a. 一天/一波节奏

```
白天战斗 → 晚上商店 → 下一天战斗 → 晚上商店 → ...
```

每个完整循环称为"1 天"（对应 SAP 的 1 turn）：

| 阶段 | 操作 | 说明 |
|---|---|---|
| 战斗阶段 (PLAYER/MONSTER) | 玩家回合 → 结束回合 → 怪物回合 | 现有逻辑不动 |
| 战斗结算 | 击杀怪物、波次结算 | 现有 `finishMonsters()` 扩展 |
| 商店阶段 (SHOP) | 获得收入 → 刷新/冻结/买/卖/合成 → 结束商店 | **新建** |
| 进入下一天 | 商店关闭，`spawnWave` 下一波 | 现有逻辑扩展 |

### 1b. 商店打开时

沿用 SAP `open_shop()` 的时序：

1. `G.phase = 'SHOP'`
2. 发放每日/每波固定金币收入
3. 根据当前天数/波次计算商店 tier
4. 按 tier 和 packs 生成商品列表（units + 强化品）
5. 上一天冻结的商品保留在对应位置
6. 恢复阵容状态，移除临时效果

### 1c. 商店关闭时

沿用 SAP `close_shop()` 的时序：

1. 触发 END_TURN / 商店关闭效果
2. 保存阵容快照
3. 金币重置为 `DEFAULT_COIN_COUNT + saved_coins`
4. `G.phase` 切回战斗相关阶段
5. 进入下一波怪物

### 1d. 金币流转

```
每天固定收入：10 金币（参考 SAP DEFAULT_COIN_COUNT）
+ 击杀怪物额外金币（每只 1-2 金币，上限控制）
+ 出售单位返还金币（出售价 = 单位等级数）
+ saved_coins（跨天保留金币）
- 购买单位（2-5 金币，随 tier 变化）
- 购买强化品（1-3 金币）
- 刷新商店（1 金币/次）
= 剩余金币 → 进入下一天战斗
```

### 1e. 刷新和冻结

直接沿用 SAP 规则：

- **刷新 (`roll`)**: 消耗 1 金币，全部非冻结商品被替换为新商品
- **冻结 (`freeze`)**: 点击商品切换冻结态，冻结商品下次刷新和关店/开店后仍保留
- 刷新时冻结商品位置不变，其他槽位重新生成

## 2. 商品类型设计

### 第一版（P0）必做

| 商品类型 | 映射自 SAP | 说明 |
|---|---|---|
| **单位卡** | Pet | 购买后放入上阵区或备战区，每个单位自带 3 个固定行动槽 |
| **元素强化品** | Food（简化版） | 对已上阵单位使用，改变某个行动槽的元素类型或增加元素层数 |
| **金币增益品** | Food 中的经济类 | 使用后增加 saved_coins 或下次收入 |

### 第二版（P1）扩展

| 商品类型 | 说明 |
|---|---|
| 行动槽升级品 | 提升某个行动槽的伤害档位（tier+1） |
| 范围扩展品 | 临时扩大某个单位的行动槽攻击形状 |
| 元素叠加品 | 给某个行动槽附加第二元素（水+火双元素槽） |

### 后续（P2/P3）再考虑

- 独立攻击方块（作为装备给单位替换某个槽）
- 遗物（全局被动效果）
- 单位皮肤/变异
- 单位羁绊
- 多商品池（packs）

### 关键约束

- 第一版不做大量独立攻击方块堆砌
- 单位就是行动槽的容器，每个单位自带 3 个固定槽
- 强化品只改槽的属性，不新增槽

## 3. 单位设计

### 3a. 单位卡结构

每张单位卡包含：

```
{
  id: string,           // 唯一标识
  name: string,         // 单位名称
  element: string,      // 元素倾向 (fire/water/wind/earth)
  tier: number,         // 出现商店 tier
  cost: number,         // 购买金币
  hp: number,           // 生命值
  maxHp: number,
  atk: number,          // 被怪物攻击时的反击/基础攻击力（可选）
  actionSlots: [        // 3 个固定行动槽
    { el, sn, dir, tier },
    { el, sn, dir, tier },
    { el, sn, dir, tier }
  ],
  level: 1,             // 当前等级
  ability: null,        // 特殊能力（2级/3级解锁）
}
```

### 3b. 第一版单位池

建议 8-12 个单位，覆盖不同打法：

#### Tier 1（初期，cost 2-3）

| 名称 | 元素 | 槽1 | 槽2 | 槽3 | 打法倾向 | 1→2级变化 | 2→3级变化 |
|---|---|---|---|---|---|---|---|
| 火苗使 | fire | 1格点·火 | 2格短线·火 | 3格直线·火 | 清怪/单体火力 | 槽1→2格短线 | 槽3→4格直线 |
| 水滴使 | water | 1格点·水 | 3格拐角·水 | 十字5格·水 | 控制/范围铺水 | 槽2→2x2方块 | 槽3伤害×1.5 |
| 微风使 | wind | 2格短线·风 | T形4格·风 | 5格直线·风 | 远程/直线穿透 | 槽1→3格直线 | 全槽范围+1格 |
| 岩盾使 | earth | 3格拐角·土 | 2x2方块·土 | L形4格·土 | 防御/地形构筑 | HP+5 | 槽3→长L形5格 |
| 均衡使 | fire/water | 1格点·火 | 1格点·水 | 3格拐角·风 | 灵活/多元素 | 全槽层数+1 | 槽1+槽2→2格短线 |

#### Tier 2（中期，cost 3-4）

| 名称 | 元素 | 槽1 | 槽2 | 槽3 | 打法倾向 |
|---|---|---|---|---|---|
| 烈焰使 | fire | 3格直线·火 | 十字5格·火 | 分叉5格·火 | 大范围引爆 |
| 洪流使 | water | 2x2方块·水 | T形5格·水 | 6格直线·水 | 铺场/连锁引爆 |
| 飓风使 | wind | 5格直线·风 | 阶梯5格·风 | 凹槽5格·风 | 多方向攻击 |
| 山岭使 | earth | L形4格·土 | 长L形5格·土 | U形5格·土 | 坚壁/阻挡 |

#### Tier 3（后期，cost 4-5）

| 名称 | 元素 | 特点 |
|---|---|---|
| 凤凰使 | fire | 死亡时十字爆炸，3级复活一次 |
| 海啸使 | water | 全元素层数+1被动光环 |
| 风暴使 | wind | 行动槽全部多1格攻击范围 |
| 大地使 | earth | 全单位HP+5被动光环 |

### 3c. 两个单位组合示例

| 组合 | 前排 | 后排 | 打法 |
|---|---|---|---|
| 火+水 | 岩盾使（铺土地形） | 火苗使（引爆） | 地形+引爆combo |
| 双火 | 火苗使 | 烈焰使 | 暴力引爆流 |
| 风+水 | 微风使（远程） | 水滴使（铺水） | 远程+控制 |
| 均衡+任意 | 均衡使（多元素） | 针对性补位 | 灵活应对 |

### 3d. 单位升级后的行动槽增强

以火苗使为例：

| 等级 | 槽1 | 槽2 | 槽3 | HP |
|---|---|---|---|---|
| 1级 | 1格点·火(tier1) | 2格短线·火(tier1) | 3格直线·火(tier1) | 20 |
| 2级 | 2格短线·火(tier2) | 2格短线·火(tier2) | 3格直线·火(tier2) | 25 |
| 3级 | 2格短线·火(tier3) | 3格直线·火(tier3) | 4格直线·火(tier3) | 30 |

## 4. 合成升级规则

直接沿用 SAP `merge_behavior` 逻辑：

### 4a. 合成触发

- 在商店界面，将同名单位拖到/移动到同名单位所在位置
- 或者：购买同名单位时，如果已有该单位在上阵/备战区，自动触发合成到已有单位

### 4b. 合成效果

```
from_unit + to_unit → to_unit 升级
```

- `to_unit.level += 1`（最高 3 级）
- 升级后触发 `onLevelUp` 效果：
  - 行动槽升级（形状变大/伤害倍率提升）
  - HP 上限提升
  - 2级/3级解锁特殊能力（如有）
- 如果 from_unit 已持有强化品，强化品效果转移到 to_unit

### 4c. 3 个 1 级 = 1 个 3 级

沿用 SAP 经验值系统：
- 1级 → 2级：需要 1 个同名单位（或 1 经验）
- 2级 → 3级：需要 2 个同名单位（或 2 经验）
- 总共需要 3 个 1 级才能合到 3 级

### 4d. 合成时商店自动补位

沿用 SAP `add_levelup_pet()`：
- 每次单位升级，商店自动刷新一个比当前 tier 高 1 级的单位商品
- 如果商店已满则不追加

## 5. 经济设计

### 5a. 第一版经济模型

| 项目 | 数值 | 说明 |
|---|---|---|
| 每日固定收入 | 10 金币 | 保证基础购买力，参考 SAP |
| 击杀怪物额外收入 | 1-2 金币/只 | 上限 5 金币/天，防滚雪球 |
| 波次通关奖励 | 3 金币 | 继承现有 `finishMonsters()` |
| 商店刷新费用 | 1 金币/次 | 同 SAP |
| 冻结费用 | 0（免费） | 同 SAP |
| 单位价格 | 2-5 金币（=cost） | 随 tier 变化 |
| 强化品价格 | 1-3 金币 | 随效果强度变化 |
| 出售返还 | = 单位等级数 | 1级返1，2级返2，3级返3 |
| saved_coins | 上限 3 金币 | 跨天保留，防过度储蓄 |

### 5b. 收入控制原则

- **固定收入充足**：10 金币/天足够买 2-3 个 tier1 单位或 1 个 tier2 + 强化
- **击杀奖励有上限**：每天最多 5 额外金币，强者小爽但不滚雪球
- **出售返还低**：只返等级数，防止反复买卖套利
- **刷新有代价**：每次 1 金币，防止无限刷找最优解
- **saved_coins 有限**：最多跨天存 3 金币，差玩家不会被碾压

### 5c. 经济商品

第一版只做一个：

- **金币袋**：cost 2 金币，使用后 saved_coins +2（下一天多 2 金币）

后续扩展：
- 利息金币（saved_coins * 1.5）
- 免费刷新券

## 6. 和战斗系统的连接

### 6a. 单位 = 行动槽容器

目前游戏结构：
```
G.heroes = { ha: {...}, hb: {...} }
G.slots = [6个槽, 每个含 hid/el/sn/tier/dir/used]
```

改为单位制后：
```
G.units = {
  active: [unitId, unitId],   // 最多 2 个上阵单位
  bench: [unitId, ...],       // 备战单位（未上阵）
}

G.heroes = {
  // 由 active 单位动态生成
  slot0_unitId: { id, name, hp, maxHp, pos, unitId },
  slot1_unitId: { id, name, hp, maxHp, pos, unitId },
}

G.slots = [
  // 由上阵单位的 actionSlots 拼接
  // unit0.slot0, unit0.slot1, unit0.slot2,
  // unit1.slot0, unit1.slot1, unit1.slot2
]
```

### 6b. 两个单位 → 6 个行动槽

```
活跃单位A: [槽A0, 槽A1, 槽A2]
活跃单位B: [槽B0, 槽B1, 槽B2]
→ 总行动槽: [0, 1, 2, 3, 4, 5]
→ 槽0-2 归单位A，槽3-5 归单位B
→ 每个单位的槽由各自的 pos 决定攻击原点
```

### 6c. 单位升级如何影响战斗

- 升级后行动槽的 sn（形状）可能变大，tier（伤害倍率）可能升高
- HP 上限提升
- 特殊能力解锁（如凤凰使 3 级复活）
- 这些变化在 `open_shop` 后、`close_shop` 前写入 units 定义，战斗时自动生效

### 6d. 元素强化品如何影响战斗

强化品对单位使用后，修改该单位某个行动槽的属性：
- 改 el（元素类型）：如把槽1从 fire 改为 water
- 加层数：该槽使用时直接附加 N 层元素到场
- 提 tier：伤害倍率 +1

这些修改在战斗阶段通过 `G.slots` 配置自动反映，不改变核心战斗结算逻辑。

### 6e. 遗物未来接入点

遗物作为全局被动效果，挂载在 `G.relics` 数组上：
- 在每个 `dispatchGameAction` 后检查触发条件
- 在 `settleExplosions` 前后检查触发条件
- 在怪物攻击前后检查触发条件
- 第一版不做

## 7. UI 流程

### 7a. 进入商店界面

```
┌─────────────────────────────────┐
│ 🛒 商店 · 第1天                 │
│ 💰 金币：10    📦 下天保留：0   │
│                                 │
│ ┌─ 商品区（单位）─────────────┐ │
│ │ [火苗使 2💰] [水滴使 2💰]  │ │
│ │ [微风使 3💰] [岩盾使 3💰]  │ │
│ │ [均衡使 3💰] [空位]        │ │
│ └────────────────────────────┘ │
│                                 │
│ ┌─ 商品区（强化品）───────────┐ │
│ │ [火元素+1 1💰] [金币袋 2💰]│ │
│ │ [空位]        [空位]        │ │
│ └────────────────────────────┘ │
│                                 │
│ ┌─ 你的阵容 ─────────────────┐ │
│ │ [上阵位1: 空] [上阵位2: 空]│ │
│ │ 备战区: (无)                │ │
│ └────────────────────────────┘ │
│                                 │
│ [🔄 刷新 1💰] [❄️ 冻结]        │
│ [⚔️ 结束商店 →]               │
└─────────────────────────────────┘
```

### 7b. 点击商品

- **点击单位卡**：弹出详情面板
  - 单位名称、元素倾向、HP
  - 3 个行动槽预览（形状图 + 元素 + tier）
  - 1/2/3 级变化预览
  - [购买 2💰] 按钮
- **点击强化品**：弹出详情面板
  - 效果说明
  - [购买 1💰] 按钮

### 7c. 购买单位后

- 如果上阵位有空（<2）：直接放入空上阵位
- 如果上阵位已满（=2）：放入备战区
- 如果已有同名单位：弹出选择 [合成到已有单位] 或 [放入备战区]

### 7d. 上阵/替换/合成/出售

- **上阵**：从备战区拖到空上阵位，或点击单位 → [上阵]
- **替换**：从上阵位拖到备战区，备战单位自动补位；或两个上阵单位交换位置
- **合成**：拖同名单位到同名单位 → 目标单位升级，素材消失
- **出售**：点击单位 → [出售 返N💰]

### 7e. 刷新和冻结

- **[🔄 刷新 1💰]**：消耗 1 金币，非冻结商品全部重随机
- **冻结**：点击商品 → 商品标记为 ❄️ 冻结态，刷新时保留
- **[⚔️ 结束商店]**：关闭商店，进入下一天战斗

## 8. 第一版最小可做范围

### 做这些

1. **商店界面**：改造现有 `.stt` 商店区，展示单位商品 + 强化品商品 + 阵容管理
2. **金币系统**：10 金币固定收入 + 现有击杀/波次奖励 + saved_coins（上限 3）
3. **刷新**：1 金币刷新，非冻结商品重随
4. **冻结**：点击商品切换冻结态，刷新时保留
5. **买单位**：从商品区购买单位，放入上阵位或备战区
6. **出售单位**：出售返等级数金币
7. **两个上阵位**：最多 2 个单位同时上阵
8. **备战区**：未上阵单位存放
9. **单位 1/2/3 级**：同名合成升级，升级后行动槽增强
10. **每个单位固定 3 个行动槽**：2 单位 × 3 槽 = 6 行动槽
11. **结束商店后进入战斗**：现有 `spawnWave` + 战斗流程
12. **第一版单位池**：5-6 个 tier1 单位 + 3-4 个 tier2 单位 + 0 个 tier3（后续加）
13. **简单强化品**：元素+1、提伤害档位、金币袋

### 不做这些

- 复杂遗物系统
- 大量独立攻击方块商店
- 装备系统
- 随机事件
- 复杂经济流派（利息、投资、贷款）
- 多单位羁绊（同元素加成等）
- 复杂 AI 推荐购买/自动选择
- 多商品池（packs）
- Tier 3 以上单位（第一版做到 tier 2 封顶）
- 单位特殊被动能力（第一版单位只有行动槽区别）

### 9e. 从 ownedUnits 生成 heroes 和 slots

```javascript
function buildHeroesFromUnits(G) {
  const active = G.ownedUnits.filter(u => u.active).slice(0, 2);
  G.heroes = {};
  G.slots = [];
  active.forEach((unit, ui) => {
    const def = UNIT_DEFS[unit.defId];
    const lvlData = def.levels[unit.level];
    const hid = `u${ui}`;
    G.heroes[hid] = {
      id: hid,
      name: def.name,
      hp: lvlData.hp,
      maxHp: lvlData.hp,
      pos: unit.pos,
      unitId: unit.instanceId,
    };
    lvlData.slots.forEach((slotDef, si) => {
      G.slots.push({
        id: ui * 3 + si,
        el: slotDef.el,
        sn: slotDef.sn,
        tier: slotDef.tier,
        dir: slotDef.dir,
        hid: hid,
        used: false,
      });
    });
  });
}
```

## 10. 风险和取舍

### 容易做复杂的机制

1. **单位被动能力**：每个单位 2 级/3 级解锁特殊效果，很容易变成"每个单位都要设计 3 层效果 × 12 个单位 = 36 个特殊规则"，第一版先只做行动槽增强
2. **强化品组合**：如果强化品可以无限叠加在同一个槽上，会出现堆满一个槽秒杀一切的情况，需要每个槽最多 1-2 个强化品
3. **合成经验值**：3 个 1 级 = 1 个 3 级的规则看起来简单，但加上出售、升级后的价格变化，经济平衡会很快变复杂

### 第一版应该砍掉的

- 遗物：全局被动效果和触发条件很多，容易引入隐蔽 bug
- 独立攻击方块商店：会和单位自带行动槽产生"槽从哪来"的冲突
- 单位羁绊：同元素加成等，第一版数据量不够，强行做会很单薄
- 复杂出售策略：第一版出售 = 固定返等级数，不做"随市场波动"等复杂规则
- Tier 3+ 商品池：第一版 battle test 最多到 tier 2

### 最影响乐趣的机制（优先级排序）

1. **买单位 → 看到新单位上阵**（最核心，必须做好）
2. **同名合成升级 → 单位变强**（爽点明确，第二优先）
3. **刷新冻结 → 追想要的单位**（策略感，第三优先）
4. **两个单位 6 个槽的组合**（策略深度，第四优先）
5. **强化品改槽属性**（锦上添花，第五优先）

### 最容易和现有战斗代码冲突的地方

| 冲突点 | 说明 | 建议 |
|---|---|---|
| `G.heroes` 结构 | 目前 `ha/hb` 写死，单位制需要动态生成 | 新增 `buildHeroesFromUnits()` 桥接函数，不改现有 core 逻辑 |
| `G.slots` 初始化 | 目前 `initGame()` 写死 6 个槽的 el/sn/hid | `closeShop` 时动态重建 slots，initGame 保留教程默认 |
| `moveHero()` | 目前两个英雄各自移动 | 单位制下仍然两个英雄，移动逻辑不变 |
| `finishMonsters()` | 目前直接进 SHOP | 改为调用 `openShop()` |
| `spawnWave()` | 目前基于 wave/round | 扩展为基于 day |
| 测试用例 | 很多测试依赖 `ha/hb` 结构 | 新测试覆盖桥接函数，旧测试尽量不改 |

### 推荐第一版落地方案

**只做"单位商店 + 合成升级"这一条线**：

- 6 个 tier1 单位 + 4 个 tier2 单位
- 每个单位 3 个固定行动槽
- 商店买单位 → 上阵 2 个 → 同名合成升级
- 升级后行动槽增强（形状/tier，不做特殊能力）
- 强化品先只做"元素+1"和"金币袋"两种
- 商店 UI 最小化改造，不新增独立界面框架

不做：遗物、装备、羁绊、复杂强化品、多商品池。
