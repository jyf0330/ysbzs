## 结论（拍板）

当前最缺的不是更多 AI Agent，而是**「小团队岗位能力包」**。

最值得先补的六块：

1. **战斗 replay 系统**（优先级最高）
2. **规则测试库**
3. **Playwright 画面验收**
4. **Debug 面板**
5. **数据表 + JSON Schema**
6. **自动打包 / 上传（Demo 后）**

目标生产线：

```text
规则 JSON → 自动测试 → 一整局 replay → Debug 面板 → Playwright 截图 → itch.io 自动发布 → 玩家反馈表 → 回来改数值
```

这条线跑通后，一人可覆盖小团队约 60% 能力：**策划验证、程序回归、QA、发行、数据反馈**。

## 优先补强六件套（详细）

### 1. 战斗 replay 系统 — 10/10 · 必须

把每一局保存成 JSON，供 AI、开发者、测试脚本复现同一局。

**建议格式：**

```json
{
  "version": 1,
  "seed": 123,
  "day": 1,
  "heroes": [],
  "monsters": [],
  "actions": [],
  "turns": [],
  "finalResult": {}
}
```

**字段说明：**

| 字段 | 内容 |
|---|---|
| `seed` | 随机种子（若有） |
| `day` | 当前天数 |
| `heroes` / `monsters` | 开战快照 |
| `actions` | 玩家输入序列（对齐 `dispatchGameAction`） |
| `turns` | 每回合结算快照（可选，便于 diff） |
| `finalResult` | 胜败、城堡 HP、引擎层数等 |

**作用：** 补 QA + 战斗策划 + 程序调试；出 bug 不靠截图猜。

**与现有代码：** 可基于 `G.actionLog` 扩展导出；回放驱动 `applyActionToState` 序列。

**验收：** 给定 replay JSON，`node test.js --replay <file>` 可复现相同 `finalResult`。

---

### 2. 规则测试库 — 10/10 · 必须

核心规则被改乱是最大风险。先锁这些用例（文件名可按 Vitest 或继续扩展现有 `test.js`）：

| 测试文件 | 验证内容 | 现状 |
|---|---|---|
| `fire-damage.spec` | 火伤害、叠层、引爆 | test.js 有部分 fire 用例 |
| `water-heal.spec` | 水治疗目标、溢出、优先级 | 待补 / 与文档火教学冲突 `[NEEDS_REVIEW]` |
| `wind-control.spec` | 风牵制、推拉、减速 | 待补 |
| `earth-block.spec` | 土阻挡、怪物撞土、消耗 | 待补 |
| `full-battle-day1.spec` | 第 1 天完整战斗 | 可参考 `playable_day1.js` |
| `full-battle-day3.spec` | 第 3 天压力战斗 | 待补 |

**工具选型：**

- **Vitest**：适合未来若拆 Vite 模块；复用 Vite 配置，跑纯逻辑测试快。
- **当前**：继续 `node test.js` 也可；迁移 Vitest 非阻塞，可按模块逐步拆文件。

**验收：** 六类 spec 均有 RED/GREEN 记录；CI 或本地一条命令全绿。

---

### 3. Playwright 画面验收 — 9/10 · 必须

补「小团队测试同事」。每次改完自动打开游戏，检查：

| 检查项 | 例子 |
|---|---|
| 棋盘没错位 | 13×13（或当前棋盘尺寸）正常 |
| 血条没飞出格 | 怪物 / 英雄血条位置 |
| 行动槽正常 | 火 / 水 / 风 / 土图标 |
| 日志正常 | 无 NaN、undefined |
| 战斗能走完 | 点击开始后不崩 |

**工具：** Playwright `toHaveScreenshot()` 做 UI 回归。

**与项目：** 版本发布验收清单已要求；I 组 UI 测试曾因无 `render_game_to_text` 暂缓 — 可优先补最小 smoke：打开 `index.html` → 开始 Day1 → 截图棋盘。

---

### 4. Debug 面板 — 9/10 · 要

开发者专用 overlay（`?debug=1` 或快捷键），比换引擎更重要：

| 面板项 | 显示内容 |
|---|---|
| 当前格子 | 坐标、单位、元素层数 |
| 元素层 | 火 / 水 / 风 / 土数量 |
| 伤害预览 | 本回合预计伤害（可读 `coreSnapshot`） |
| 怪物路径 | 下一步走哪 |
| 行动队列 | 4/6 槽依次执行 |
| 结算日志 | 每一步状态变化（`actionLog` 尾部） |

**与现有代码：** `recomputeCorePreview()` / `buildPreviewGrid()` 已有预览数据，面板主要是展示层。

## 其他岗位怎么补

### 美术

**推荐：** Krita + Penpot / Figma + 统一资产命名表。

| 资产规范 | 例子 |
|---|---|
| 命名 | `hero_fireling_lv1_idle.png` |
| 尺寸 | 64×64 / 128×128 |
| 风格规则 | 米色底、黑线稿、少色块 |

**关联：** [`美术风格指南.md`](../03_美术资产（美术主导）/美术风格指南.md) · [`资源目录结构与命名.md`](../03_美术资产（美术主导）/资源目录结构与命名.md)

### 音效

**推荐：** Audacity + Bfxr / Sfxr。

| 音效 | 用途 |
|---|---|
| 点击 | 选英雄 / 选格子 |
| 放置 | 行动槽命中 |
| 火伤害 | 火结算 |
| 水治疗 | 治疗 |
| 土阻挡 | 挡怪 |
| 怪物受击 | HP 下降 |
| 死亡 | 怪物消失 |
| 商店购买 | 买英雄 |

### 数据分析

**推荐：** 先自建事件日志，后接 GameAnalytics。

| 事件 | 价值 |
|---|---|
| `battle_start` | 玩家开始哪一天 |
| `battle_end` | 输赢、剩余血量 |
| `hero_bought` | 哪些英雄有人买 |
| `monster_killed` | 哪些怪太弱 |
| `castle_damage` | 每天推进强度 |
| `turn_count` | 一局太长还是太短 |

第一版：本地 `localStorage` 或导出 JSON，不必急着接 SDK。

### 错误监控

**推荐：** Sentry（支持 JS source map）。**试玩版上线后再接**，不必第一天接。

### 本地化

**推荐：** 先 CSV / JSON 文本表，后 Weblate。

```json
{
  "hero.fireling.name": "火苗灵",
  "hero.fireling.desc": "向前方格子叠加火元素",
  "battle.fire_damage": "火{layers}层造成{damage}伤害"
}
```

## 关卡 / 地图工具怎么选

| 工具 | 评分 | 适合你吗 | 说明 |
|---|---:|---|---|
| **自制关卡 JSON / 表格** | 9/10 | **最适合第一版** | 规则驱动；天数 + 波次 + 英雄行为，非复杂 tilemap |
| **LDtk** | 8.5/10 | 适合 | 现代 2D 关卡编辑器，JSON 导出 |
| **Tiled** | 8/10 | 也适合 | 老牌地图编辑器，JSON 导出 |

**拍板：** 第一版先自制简单 JSON 编辑器 / 表格，不急着上 LDtk。瓶颈在战斗规则闭环，不在地图编辑。

## 不建议现在做的

| 不建议 | 原因 |
|---|---|
| 立刻换 Godot / Unity | 迁移成本大，核心规则还没稳 |
| 立刻上 Steam 全套 | Demo 没稳，只增加压力 |
| 立刻做复杂关卡编辑器 | 瓶颈是战斗规则闭环 |
| 立刻追求完整美术管线 | 拖慢玩法验证 |
| 立刻接一堆数据平台 | 先自建最小事件日志就够 |
