## 状态与数据结构

- 状态管理方式：单一全局对象 `G`，每次操作后调用 `recomputeCorePreview()` + `render()` 刷新视图。
- `G.phase`：`'PLAYER'` / `'MONSTER'` / `'SHOP'` / `'OVER'`
- `G.heroes`：`{ha: {id,name,hp,maxHp,pos,atk}, hb: {...}}`
- `G.monsters`：`[{id,name,hp,maxHp,pos,atk,el,dead,...}]`
- `G.slots`：`[{id,hid,el,sn,tier,dir,used}]`（6个：ha 用 0-2，hb 用 3-5）
- `G.board`：`13×13` 二维数组，每格 `{el,stk}`，作为当前可视元素投影；元素规则判定以 `G.elementCells` 为准
- `G.selectedCell`：`{r,c}` 或 `null`，当前选中的格子，用于打开详情面板
- `G.elementCells`：`{"r,c":{fire:{layers,...},water:{...},...}}` — 按格子存储四元素独立叠层，是元素占用、结算和移动阻挡的权威数据
- `G.explosionThreshold`：触发引爆的最小层数（默认=3）
- `G.coreSnapshot`：核心只读快照，包含 `previewGrid`、`monsterStats`、`cellInfoMap`、`monsterThreats`、`heroStats`、`battleReport`
- `G.actionLog`：事件日志，用于回放和战报
- `G.coreVersion`：快照版本号，每次重算递增

### coreSnapshot 详细结构

```js
{
  _version: Number,
  _ts: Number,
  previewGrid: {
    grid: {
      "r,c": {
        cellKey: "r,c",
```

- **事件驱动架构**：玩家操作优先进入 `dispatchGameAction(action)` → `applyActionToState()` 修改 `G` → `recomputeCorePreview()` 统一重算 → `render()` 刷新视图。
- **previewGrid 为唯一预览源**：`buildPreviewGrid()` 生成全棋盘 169 格独立预览数据；`buildBoardVM()`、`renderBoard()`、`renderCellDetail()` 三者统一从此读取，不再各自独立计算。
- **元素场为权威占用源**：英雄移动、怪物移动、怪物撞元素和商店放置都通过 `hasElementAt()` / `topElementAt()` 判断元素地形；结算和消耗后用 `syncBoardElementFromElementCells()` / `clearElementAt()` 保持 `board` 投影一致。
- **多元素同时爆炸**：同一空格多个元素均达到引爆阈值时，各自独立计算爆炸伤害并累加，预览与真实结算一致。
- **视图层不算规则**：`renderXxx()` 函数只接受 VM 或 previewGrid cell，不承担战斗规则计算。
- **双输出并行**：同一份核心状态一边生成 `G.coreSnapshot` 供分析，一边生成 `buildBattleVM()` 供界面渲染。

## 渲染层分层结构（Core → previewGrid → ViewModel → View）

视觉表现层按四层组织，所有代码仍内联在 `index.html` 中：

```
Core（游戏规则层）
  G / dispatchGameAction / applyActionToState / useSlot / monsterAct / settleExplosions / ...
        ↓ 只读
previewGrid 层（独立格子预览层）
  buildPreviewGrid()    → 全棋盘 169 格，每格 entity/elementField/preview
  getSelectedCellPreview() → 从 previewGrid.grid[key] 读详情
        ↓ 输出纯数据
ViewModel 层（buildXxxVM 系列）
  buildBoardVM()        → 从 previewGrid.grid 读取每格预览数据
  buildHeroesVM()       → 英雄面板数据
  buildSlotsVM()        → 行动槽数据
  buildTurnVM()         → 回合信息
  buildBattleVM()       → 汇总所有 VM
        ↓ 传参
View 层（renderXxx(vm) 系列）
  renderBoard(boardVM)  → 只创建 DOM
  renderCellDetail(cell) → 只创建详情 DOM，不计算规则
  renderHS(heroesVM)    → 只创建 DOM
  renderSlots(slotsVM)  → 只创建 DOM
  renderTurn(turnVM)    → 只更新文本
```

**原则**：previewGrid/ViewModel 层不创建 DOM；View 层不计算规则；CSS 类是唯一的视觉边界。

## 架构约束

- 单文件架构：所有 CSS / HTML / JS 在 `index.html` 内，不拆分。
- `renderBoard(boardVM)` 每帧完整重绘棋盘 DOM，不做增量更新。
- 怪物预览（simMonAct）和怪物实际行动（monsterAct）使用相同 3AP 逻辑，保持预览和执行一致。
- 所有格子坐标用 `"r,c"` 字符串作为 Map key。
- 点击棋盘格默认打开格子详情，行动槽只通过「使用」按钮发动。
