## 状态与数据结构

- 状态管理方式：单一全局对象 `G`，每次操作后调用 `render()` 刷新视图。
- `G.phase`：`'PLAYER'` / `'MONSTER'` / `'SHOP'` / `'OVER'`
- `G.heroes`：`{ha: {id,name,hp,maxHp,pos,atk}, hb: {...}}`
- `G.monsters`：`[{id,name,hp,maxHp,pos,atk,el,dead,...}]`
- `G.slots`：`[{id,hid,el,sn,tier,dir,used}]`（6个：ha 用 0-2，hb 用 3-5）
- `G.board`：`13×13` 二维数组，每格 `{el,stk}`
- `G.hitCount`：连击计数（影响伤害倍率）
- `G.elementCells`：`{"r,c":{fire:{layers,previewDamage,willExplode},...}}` — 按格子存储四元素独立叠层指标
- `G.explosionThreshold`：触发 willExplode 的最小层数（默认=1）
- `G.previewEvents`：`[{el,r,c,dmg}]` — 仅用于飘字显示，不做真实才怯屯血

## 渲染层分层结构（Core → ViewModel → View）

视觉表现层按三层组织，所有代码仍内联在 `index.html` 中：

```
Core（游戏规则层）
  G / addEl / useSlot / monsterAct / explode / ...
        ↓ 只读
Preview 计算层（提取自 renderBoard）
  computeHeroAttackPreview()  → heroAtkMap
  computeMonsterActionPreview() → {monActMap, heroIncomingDmg}
        ↓ 输出纯数据
ViewModel 层（buildXxxVM 系列）
  buildBoardVM()   → 13×13 cells，含 hero/mon/el/classes/pvEl/monDmg/...
  buildHeroesVM()  → 英雄面板数据
  buildSlotsVM()   → 行动槽数据
  buildTurnVM()    → 回合信息
  buildBattleVM()  → 汇总所有 VM
        ↓ 传参
View 层（renderXxx(vm) 系列）
  renderBoard(boardVM)  → 只创建 DOM
  renderHS(heroesVM)    → 只创建 DOM
  renderSlots(slotsVM)  → 只创建 DOM
  renderTurn(turnVM)    → 只更新文本
```

**原则**：Preview/ViewModel 层不创建 DOM；View 层不计算规则；CSS 类是唯一的视觉边界。

## 架构约束

- 单文件架构：所有 CSS / HTML / JS 在 `index.html` 内，不拆分。
- `renderBoard(boardVM)` 每帧完整重绘棋盘 DOM，不做增量更新。
- 怪物预览（simMonAct）和怪物实际行动（monsterAct）使用相同 3AP 逻辑，保持预览和执行一致。
- 所有格子坐标用 `"r,c"` 字符串作为 Map key。
