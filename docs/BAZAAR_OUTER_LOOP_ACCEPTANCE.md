# 大巴扎式外层系统长期验收标准

本文是后续持续开发的验收锚点。目标不是做一个战斗游戏外面套菜单，而是做一个《The Bazaar》式外层构筑 run；其中战斗节点进入元素背包史自己的棋盘元素战斗。

## 总目标

玩家完成一局 run 时，外层体验必须满足：

```text
每天做构筑选择
-> 经过商人 / 摊位 / 奖励 / 事件 / 遭遇
-> 形成可追踪的经济、背包、宠物、元素和流派成长
-> 进入元素背包史棋盘战斗验证构筑
-> 根据胜负、奖励和下一天压力继续推进
-> 直到最终 Boss 或明确失败
```

## 不可变边界

1. 外层对齐《The Bazaar》的 run 结构、节点节奏、商人摊位、奖励选择、经济压力和构筑成长。
2. 内层战斗不照搬《The Bazaar》，必须继续使用当前元素背包史棋盘战斗、宠物行动、元素层、预览和结算。
3. UI 只能通过 `/api/action` 发玩家意图，通过 `/api/view` 读 ViewModel，通过 `/api/report` 读战报。
4. 核心层不得直接操作 DOM；所有规则状态变化必须走 reducer / core。
5. 后续任何局部任务，如果不能说明它推进了本文哪一条验收标准，默认不是主线优先级。

## 第一层：外层 Run 骨架验收

通过标准：

- 至少支持 Day1-Day10 或等价完整 run 长度。
- 每天有明确日程结构，不是单一“进商店 -> 打架”循环。
- 日程至少包含：成长节点、商人/摊位节点、奖励节点、遭遇选择、固定战或 Boss 战。
- 每个可选节点至少 3 选 1；固定节点必须解释为什么不可选。
- 节点选择写入结构化历史，ViewModel 能展示当天、当前步骤、已选节点、下一步。
- 运行脚本或测试能从 Day1 自动推进到终局，并能断言最终胜利、失败或到达终局状态。

未通过信号：

- 只能跑 Day1。
- 后续天数只是复制战斗，没有外层选择。
- 节点选择只改 UI 文案，不改变核心状态。
- 选择历史无法回放或无法从 ViewModel 读取。

## 第二层：商人 / 摊位 / 标签池验收

通过标准：

- 商店不是一个固定随机池，必须支持多个商人或摊位。
- 每个商人/摊位有标签、商品池、价格规则、槽位数量和解锁天数。
- 商品生成能按标签筛选，例如元素、流派、等级、宠物类型、奖励类型。
- 玩家能看到每个摊位的身份和商品倾向，而不是只看到一排商品。
- 刷新、折扣、免费刷新、定向补货必须作为状态存在，并能被事件或奖励节点修改。
- 价格和售卖规则跟公开成长轴一致：青铜、白银、黄金、钻石是玩家可见成长轴；pT 只做内部池等级。

未通过信号：

- 商店仍是单池随机 6 格。
- 折扣和刷新只是日志文字，没有状态。
- 标签只存在 CSV，不参与生成。
- 商品等级、价格和奖励节点互相打架。

## 第三层：构筑成长验收

通过标准：

- 玩家一局内能围绕至少两个流派形成选择，例如火系、召唤、陷阱、元素转换或经济流。
- 外层节点能影响构筑：获得宠物、升级、复制、定向商品、折扣、刷新、奖励、事件效果。
- 每次选择都有短期收益和长期方向，不只是数值加减。
- 背包、队伍、金币、刷新、折扣、奖励候选都能在 ViewModel 中被读取。
- 关键成长变化进入结构化日志或 changeLog，能用于战报、回放或调试。

未通过信号：

- 玩家只是在战斗间随机拿奖励。
- 所有选择都等价，不能形成流派。
- 外层奖励无法影响下一场战斗。
- 构筑状态只能从 DOM 文案猜测，核心对象不可读。

## 第四层：战斗接入验收

通过标准：

- 外层进入战斗时，使用当前元素背包史战斗核心。
- 战斗结果能回写外层：胜负、奖励资格、金币变化、下一节点解锁、失败惩罚。
- 敌人和战斗压力来自当天、节点、遭遇或 Boss 配置，而不是硬编码单场演示。
- 玩家行为验收必须走真实入口：按钮、dispatch、autoExecuteTurn 或 `/api/action`。
- 战斗 UI 继续遵守可见验收门禁：真实浏览器操作、截图、DOM / ViewModel / console 证据、主线程复核。

未通过信号：

- 外层节点和战斗彼此独立。
- 战斗胜负不影响 run。
- 为了做外层而绕开现有战斗核心。
- 直接调用内部函数当作玩家链路验收。

## 第五层：物体触发系统验收

通过标准：

- `elementPackets`、modifier、triggerQueue、objectRegistry、changeLog 不只是单元测试样例，而要贯穿常用战斗行动。
- 触发顺序稳定：优先级、来源类型、棋盘顺序、创建时间、id 都可断言。
- 场上单位、元素包、陷阱、持续效果、遗物或外层奖励能进入统一可触发对象视图。
- 触发结果写入结构化 changeLog，并能生成玩家可读战报。
- 至少一个外层奖励或摊位效果能转化成战斗内 modifier / trigger / object。

未通过信号：

- triggerQueue 只排序，不真正执行常用行动。
- objectRegistry 只收集，不参与监听或派发。
- modifier 只在孤立测试里生效。
- 回放数据不能解释一次关键触发链。

## 第六层：可视化和玩家理解验收

通过标准：

- 第一屏能让玩家知道：今天第几天、当前节点、金币、构筑核心、下一步选择。
- 节点选择界面能清楚区分商人、摊位、奖励、事件、遭遇、Boss。
- 商店界面能看到摊位身份、商品倾向、价格、折扣、刷新状态。
- 战斗前能看到将要面对的遭遇或 Boss 压力。
- 战斗中仍以数字优先：HP、伤害、关键预览不能被单位或装饰遮挡。
- 任何 UI / 棋盘 / 可见预览改动提交前必须有真实浏览器截图证据。

未通过信号：

- 玩家不知道自己在 run 的哪一步。
- 节点、商店、战斗只是同一组按钮换文案。
- 关键数字被遮挡。
- 只有核心测试，没有真实浏览器可见验收。

## 分阶段 Definition of Done

### Phase A：Day1 样板闭环

- Day1 有完整节点链路。
- 节点 3 选 1、中午遭遇 3 选 1、晚上战斗可跑。
- ViewModel 能读 dayRoute。
- 浏览器验收截图存在。

当前状态：已基本达到。

### Phase B：Day1-Day3 可玩外层

- 连续 3 天可玩。
- 每天至少 4 个外层决策点。
- 商人/摊位标签开始实际影响商品。
- 战斗胜负回写外层经济或奖励。
- 自动脚本能跑完 Day1-Day3。

### Phase C：Day1-Day10 结构闭环

- 10 天日程全部数据化。
- Day4 起出现更高等级资源或商品，Day6 起出现更高压力奖励或 Boss 前置。
- 至少两个流派能从节点和商店中稳定成长。
- 自动脚本能跑到 Day10 终局。

### Phase D：大巴扎式商店闭环

- 多商人/摊位、标签池、价格、刷新、折扣、定向补货全部进入核心状态。
- UI 明确呈现每个摊位身份。
- 奖励和事件能改变商店状态。
- 商店选择能影响后续战斗胜率。

### Phase E：触发物体闭环

- 外层奖励能生成战斗内触发物、modifier 或持续效果。
- 常用战斗行动经过 triggerQueue 或等价统一触发层。
- 结构化日志能解释触发链。
- replay 数据足以复盘一次关键战斗。

### Phase F：发布级竖切

- 新玩家能从 Day1 玩到终局，不需要开发者解释。
- 外层 run、商店、构筑、战斗、奖励、失败/胜利全部在 UI 中闭环。
- `npm run check:all`、`npm run test:coverage`、浏览器验收全部通过。
- 任务卡、截图、DOM / ViewModel / console 证据齐全。

## 后续任务优先级规则

优先做：

1. 能把 Day1 样板扩展到更多天的任务。
2. 能把商店从单池改成商人/摊位/标签池的任务。
3. 能让外层奖励真正影响战斗的任务。
4. 能让触发系统从 MVP 进入常用行动链路的任务。
5. 能让玩家看懂 run 状态和选择后果的 UI 任务。

降低优先级：

- 只优化单场战斗、但不影响外层 run 的任务。
- 只改 UI 装饰、不提高玩家理解的任务。
- 只加孤立机制、不能进入节点/商店/战斗闭环的任务。
- 只写文档、不产生可验证脚本或任务拆分的任务。

## 每个后续任务的验收模板

每张任务卡必须回答：

```text
本任务推进本文哪一层验收：
外层状态字段：
玩家入口：
ViewModel / report 证据：
自动测试：
如果可见，浏览器截图：
未完成风险：
```

如果回答不出来，先不要实现，先补设计或拆任务。
