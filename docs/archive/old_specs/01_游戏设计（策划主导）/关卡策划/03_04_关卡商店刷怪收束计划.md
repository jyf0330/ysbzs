| 主题 | 当前正式口径 |
|---|---|
| 垂直切片 | 水+召唤引擎，死亡留水层；爆爆灵等能力可额外铺火 |

## 二、03 文件组收束

### 2.1 保留内容

`03_10天商店闭环验证_大巴扎对齐版.md` 中可保留的设计资产：

- 大巴扎式多节点成长：午间商人、夜晚商店、精英奖励、Boss 奖励、特殊事件。
- 商店只卖英雄的构筑方向。
- 多摊位 / 标签池 / 同名合成 / 刷新递增 / 冻结等借鉴点。
- Boss/精英奖励英雄化，而不是给独立遗物。
- pT3/pT4 作为远期英雄池扩容方向。

`03_步骤456_定价新英雄多目标.md` 中可保留的设计资产：

- 价格函数与英雄定义应从硬编码拆成可读配置。
- 火/水/召唤英雄需要统一命名和职责表。
- 英雄的攻击形状、元素、品级、构筑标签需要集中维护。

### 2.2 不能直接采用的内容

| 内容 | 原因 |
|---|---|
| 10 天游玩日历直接实现 | 当前正式 Run GDD 是 5 天软上限 |
| pT4 / final_boss 直接进当前 S3 | 属于远期内容量，超出当前 S3 P0/P1 |

| “总计约 32 行，不涉及新系统” | 午间/精英/Boss/事件节点实际涉及奖励分发、商店生成和测试，不是小补丁 |

## 三、04 文件组收束

### 3.1 保留内容

`04_第2天代码差距.md` 的价值不再是“代码缺口”，而是 Day2 策划意图：

- Day2 要验证厚皮怪压力。

- 怪物基础表：normal、thick、fast、heavy、swarm、elite、blocker、siege、boss5、boss8、boss10 等。
- 先基础数值、后 ability 的分层方式。
- ability hook 表可以作为未来统一能力系统的接口草案。
- 压力曲线表可以作为后续数值验收工具，而不是当前实现清单。

## 六、后续可执行任务拆分

### Task A：Day2-5 当前闭环刷怪复核

目标：只检查当前 5 天 `DAY_WAVE_CONFIG` / `DAY_ROUND_CONFIG` 与正式 GDD 是否一致。

允许改动：测试和小范围波次配置。

禁止：Day6-10、pT4、final_boss、完整 ability system。

### Task B：商店节点奖励规格

目标：把午间商人、精英奖励、Boss 奖励收束成当前 5 天可用规则。

前置：定价模型和经济模型拍板。

### Task C：价格模型决策落地

目标：决定 `priceTier` 是否参与价格。若不参与，改名为 `powerTier` / `poolTier` 或在文档中明确仅作筛选权重。

前置：用户拍板。

### Task D：怪物类型表数据层

目标：将当前 Day1-5 已用怪物抽成更清晰的 `MONSTER_TYPES` 数据表。

禁止：同时接入完整 ability system。

### Task E：Ability system 单 hook 试点

目标：只选一个低风险 hook，例如 `onDeath` 或 `onSpawn`，用一个怪物/英雄能力验证接口。

前置：Day1-5 闭环和测试稳定。

### Task F：召唤物 AI 行为补齐

目标：验证是否需要从“相邻攻击”升级为“移动追怪 + 攻击”，并处理跨天清理规则。

前置：召唤元素口径和 Day2-5 关卡目标拍板。

### Task G：pT3/pT4 英雄与入池

目标：补黄金/钻石英雄、同名保证位、合成路径和商店入池规则。

前置：run 长度和定价模型拍板。

## 七、审核结论

03/04 文件组可以作为高价值设计素材，但不能作为当前代码任务直接执行。

当前最稳妥路径是：

1. 先完成本文档收束。
2. 由用户拍板“5 天继续 / 10 天扩容”和“定价模型”。
3. 再按 Task A-G 分小任务 TDD 实现。

在拍板前，任何执行 AI 都应把本文件视为 03/04 的入口，而不是直接读取旧草案开工。
