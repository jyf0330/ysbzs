# 核心层与展示层分离设计 · 元素背包史

## 文档状态

- 状态：草稿（从 `index.html` 与技术架构总览推断）
- 最后更新：2026-05-25

## 初始化结论

- 分层目标：在保持单文件架构的前提下，把游戏规则、预览计算、ViewModel、DOM 渲染分清职责。
- 当前阶段：脚本原型 / 中间层稳定之间。
- 是否允许展示层临时承载部分规则：允许原型期临时承载，但新改动应优先沉到 Core 或 ViewModel 边界 `[NEEDS_REVIEW]`。

## 核心层职责清单

- 游戏状态：`G` 全局对象，包含 phase、heroes、monsters、slots、board、elementCells、previewEvents 等。
- 战斗规则：行动槽使用、元素叠层、引爆阈值、怪物 3AP 行动、回合推进。
- 数值规则：直接命中、元素叠层、previewDamage、settleExplosions 真实扣血。
- 商店 / 背包：购买、装备、合成升级。
- 可测试边界：`test.js` 通过 Node.js 抽取 / 模拟核心函数，当前基准以项目测试输出为准。

## 展示层职责清单

- HUD / 面板 / 棋盘 DOM：由 renderBoard、renderHS、renderSlots、renderTurn 等函数创建或更新。
- 视觉状态：CSS 类、徽章、颜色、警告样式、选中态和禁用态。
- 临时提示：Tooltip、消息弹窗、商店遮罩、行动点分配弹窗。
- 操作反馈：按钮状态、格子高亮、日志滚动、预览刷新。

## 中间层职责清单

- `computeHeroAttackPreview()`：只读核心状态，生成英雄攻击 / 元素引爆预览。
- `computeMonsterActionPreview()`：只读核心状态，生成怪物行动预警和英雄预期承伤。
- `buildBoardVM()` / `buildHeroesVM()` / `buildSlotsVM()` / `buildTurnVM()`：把核心状态转换为展示层可直接渲染的数据。
- `buildBattleVM()`：汇总当前战斗界面 ViewModel。

## 核心与展示接口边界

| 边界 | 当前约定 |
|---|---|
| 核心输出 | `G` 与 ViewModel 构建函数输出的纯数据 |
| 展示读接口 | render 函数读取 ViewModel，不重新计算规则 |
| 展示写入口 | 点击 / 按钮事件调用核心动作函数，如 useSlot、endPlayerTurn、buyItem |
| 禁止事项 | View 层不直接改写怪物 HP、元素层数、回合阶段等核心状态 |
| 兼容例外 | 历史代码中的直接 DOM 事件与核心动作仍混写，后续小步收敛 `[NEEDS_REVIEW]` |

## 核心实现策略

- 原型期：继续保持 `index.html` 单文件，优先验证玩法。
- 稳定期：固定 Core → ViewModel → View 边界，新增功能先定义数据输出，再渲染。
- 长期复用期：当规则稳定、回放 / 存档边界明确后，再评估是否下沉到独立核心或 C# 核心。
