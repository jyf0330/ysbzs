# 玩家谜题投稿工具设计稿

## 定位

这是一个给玩家用的 8x8 宠物战斗谜题编辑器 / 投稿器。玩家不需要懂代码、表格、YAML 或 JSON，只需要用自然语言、简单表单和棋盘摆放表达创意。AI 负责整理格式，系统负责校验谜题能不能进入后续验证链路。

本资料包只定义新页面和数据结构，不接入当前运行时，不修改核心战斗逻辑。

## 核心原则

| 原则 | 含义 |
|---|---|
| 玩家负责想创意 | 玩家可以乱写、随便摆，不要求理解内部数据。 |
| AI 负责整理格式 | AI 把自然语言、表单和棋盘输入整理成标准字段。 |
| 系统负责校验能不能玩 | 基础校验即时做，高级可玩性校验后续接模拟器。 |
| 不让玩家直接填复杂表 | 复杂枚举、默认值、导出结构由工具和 AI 隐藏。 |
| 不复制运行时规则 | 投稿工具只生成谜题数据，真实战斗验证以后走核心模拟器。 |

## 页面结构

页面分 6 个区域，默认从上到下、从左到右完成。桌面端建议使用“左侧表单 + 中间棋盘 + 右侧检查/导出”的工作台；移动端折叠为步骤式编辑。

| 区域 | 玩家要做的事 | 推荐控件 | 产出字段 |
|---|---|---|---|
| 谜题基础信息 | 填名称、难度、类型、回合、简短说明 | 文本框、分段控件、下拉、步进器 | `name`, `difficulty`, `type`, `turnLimit`, `description` |
| 可用宠物选择区 | 从宠物库勾选本题可用宠物，并设置最多上阵数 | 搜索、复选列表、数量步进器 | `availablePets`, `maxDeployPets` |
| 8x8 棋盘摆放区 | 点击格子放我方、敌人、Boss、元素格、陷阱或空格 | 坐标棋盘、放置调色板、格子检查器 | `placements`, `boardSize` |
| 敌人设置区 | 添加敌人名称、生命、攻击、位置、Boss 标记 | 敌人列表、表单、Boss 开关 | `enemies` |
| 胜利条件设置区 | 从模板选择目标，不写复杂规则 | 模板下拉、目标选择器、补充说明 | `winCondition` |
| 标准解法区 | 可选填写步骤和结果 | 步骤列表、结果摘要、AI 提示 | `solution` |

## 字段总表

完整字段表见同目录：

- `puzzle_submission_tool_fields.xlsx`
- `puzzle_submission_tool_fields.csv`

字段表的核心列：

| 列名 | 用途 |
|---|---|
| 区域 | 对应页面 6 个编辑区域或导出区。 |
| 字段ID | 程序内部字段名。 |
| 玩家看到的字段 | UI 上展示给玩家的标签。 |
| 数据类型 | string、enum、integer、array、object、coordinate 等。 |
| 必填 | yes、no、conditional。 |
| 默认值 | AI 或编辑器可自动补的默认值。 |
| 选项/来源 | 枚举值、宠物库、棋盘坐标或模板来源。 |
| 校验规则 | 最小硬校验或提示规则。 |
| 导出JSON路径 | 最终程序可读 JSON 中的位置。 |
| UI控件 | 推荐控件形态。 |
| 备注 | AI 整理或后续接入注意事项。 |

## YAML 结构

机器可读配置见：

- `puzzle_submission_tool_schema.yaml`

YAML 负责定义：

| 模块 | 内容 |
|---|---|
| `tool` | 工具 ID、版本、职责和运行时边界。 |
| `editor_layout` | 6 个页面区域及推荐控件。 |
| `puzzle_schema` | 标准谜题数据字段、类型和约束。 |
| `win_condition_templates` | 胜利条件模板，不让玩家写复杂规则。 |
| `validation_rules` | 硬校验和高级提示校验。 |
| `ai_normalization` | AI 从自然语言整理成标准谜题的流程。 |
| `export_targets` | 小红书可读版、JSON、AI 检查报告。 |
| `example_puzzle` | “十层火线”示例数据。 |

## AI 整理流程

玩家可以只写一句话：

```text
我想做一个 1 回合火爆发题。Boss 90 血，在第5行第6列。
可用宠物是棉悠悠、捣蛋猫、皮皮鸡、翠叶鼠、火绒狐。
最多选 4 只。目标是 1 回合打死 Boss。
```

AI 整理时按以下顺序处理：

| 步骤 | 动作 | 输出 |
|---|---|---|
| 1 | 提取意图 | 类型、回合、Boss、候选宠物、目标。 |
| 2 | 名称转 ID | 宠物 ID、敌人临时 ID、胜利条件模板 ID。 |
| 3 | 补默认值 | 缺失难度默认普通，攻击力默认 0，谜题名可自动生成。 |
| 4 | 只追问阻塞字段 | 坐标不明、目标不明、宠物名无法匹配时才问。 |
| 5 | 生成标准谜题 | 输出程序可读 JSON。 |
| 6 | 执行校验 | 输出 AI 检查报告。 |
| 7 | 渲染投稿文案 | 输出小红书可读版。 |

## 导出格式

### 小红书可读版

```text
谜题名：十层火线

难度：普通
类型：火爆发题
限制：1 回合

可用宠物：
棉悠悠、捣蛋猫、皮皮鸡、翠叶鼠、火绒狐

规则：
5 只宠物里只能选择 4 只行动。
目标是在 1 回合内击杀 90 血 Boss。

敌人：
铁壳考官，90 血，位于第5行第6列。

胜利条件：
1 回合内击杀铁壳考官。

问题：
你会选择哪 4 只宠物？
```

### 程序可读 JSON

```json
{
  "puzzleId": "puzzle_fire_001",
  "name": "十层火线",
  "difficulty": "normal",
  "type": "fire_burst",
  "boardSize": [8, 8],
  "turnLimit": 1,
  "availablePets": ["pal_001", "pal_002", "pal_003", "pal_004", "pal_005"],
  "maxDeployPets": 4,
  "enemies": [
    {
      "id": "enemy_boss_001",
      "name": "铁壳考官",
      "hp": 90,
      "atk": 0,
      "position": { "row": 5, "col": 6 },
      "isBoss": true
    }
  ],
  "winCondition": {
    "type": "kill_target",
    "targetId": "enemy_boss_001",
    "withinTurns": 1
  },
  "solutionOptional": true
}
```

### AI 检查报告

```text
检查结果：

1. 是否有明确胜利条件：通过
2. 是否有可用宠物：通过
3. 是否有敌人：通过
4. 是否能在限制回合内完成：需要验证
5. 是否可能存在多解：可能存在
6. 是否适合玩家投稿：通过

建议：
这个谜题目标清楚，适合做火爆发入门题。
如果想提高难度，可以隐藏标准答案，只给玩家棋盘和宠物列表。
```

## 必须校验

| 校验 | 类型 | 失败处理 |
|---|---|---|
| 不能把单位放到棋盘外 | 硬校验 | 阻止提交并定位格子。 |
| 不能两个单位放在同一个格子 | 硬校验 | 阻止提交并要求二选一。 |
| Boss 必须有生命值 | 硬校验 | 阻止提交并聚焦生命值输入。 |
| 胜利条件必须存在 | 硬校验 | 阻止提交并打开胜利条件区。 |
| 可用宠物不能为空 | 硬校验 | 阻止提交并打开宠物选择区。 |
| 限制回合数必须大于 0 | 硬校验 | 自动改成 1 或要求玩家确认。 |
| 击杀题必须有敌人 | 硬校验 | 阻止提交并提示添加敌人。 |
| 保护题必须指定保护对象 | 硬校验 | 阻止提交并提示选择目标。 |

## 高级检查

高级检查不阻止玩家保存草稿，但会影响“是否推荐投稿”。

| 检查 | 当前处理 | 后续接入 |
|---|---|---|
| 是否无解 | 标记“需要验证” | 接战斗模拟器或 AI 搜索器。 |
| 是否太简单 | 标记“可能过易” | 比较普通攻击伤害、元素贡献和回合余量。 |
| 是否可能多解 | 标记“可能存在” | 搜索多条行动序列。 |
| 是否需要标准答案 | 没写时提示写核心思路 | 投稿审核可要求困难题必须填写。 |
| 普攻是否直接能过 | 标记“机制可能无意义” | 接伤害预估器。 |

## 新页面实现建议

这不是本轮实现范围，但后续做页面时建议按以下方式接入：

| 层 | 建议 |
|---|---|
| UI 页面 | 只维护编辑草稿和表单状态，不直接改核心战斗 state。 |
| AI 整理 | 接收玩家自然语言和草稿，输出标准 puzzle JSON。 |
| 校验器 | 先做 schema/坐标/重复格等静态校验。 |
| 战斗验证 | 后续通过核心模拟器或 `/api/action` 玩家链路验证是否能解。 |
| 导入游戏 | 需要单独 adapter，把 puzzle JSON 转成当前核心可执行场景。 |

## 边界模糊点

以下问题后续实现前建议确认：

| 问题 | 默认建议 |
|---|---|
| 投稿工具是面向玩家公开页面，还是内部运营审核工具？ | 先按公开玩家页面设计，但导出前加 AI 检查报告。 |
| 棋盘初始我方宠物是否必须摆在棋盘上？ | 不强制，允许只给可用宠物池；需要摆位题再要求坐标。 |
| 宠物库使用当前 127 宠 ID，还是先用示例 5 宠？ | 本设计用 `existing_pet_pool` 占位，接入时映射当前宠物库。 |
| 标准答案是否公开给其他玩家？ | 默认投稿数据保存答案，但小红书可读版默认不展示答案。 |
| JSON 是否直接导入当前游戏？ | 不是直接导入；需要后续 adapter。 |
